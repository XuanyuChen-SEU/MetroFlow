# MetroFlow 项目结构设计

## 1. 设计目标

这份文档用于固定 MetroFlow 的工程目录、职责边界和文件命名，避免后续生成代码时出现：

- 前后端目录混乱
- 数据脚本和业务代码耦合
- 文档、SQL、数据产物散落
- 运行步骤和实际结构不一致

## 2. 顶层目录

```text
final-project/
├── frontend/
├── backend/
├── sql/
├── scripts/
├── data/
├── docs/
├── demo.html
└── README.md
```

各目录职责如下：

- `frontend/`：Vue3 可视化前端工程
- `backend/`：Flask API 服务
- `sql/`：数据库结构与初始化脚本
- `scripts/`：数据清洗、转换、导入脚本
- `data/`：原始数据与处理中间产物
- `docs/`：系统设计、结构说明、后续部署文档

## 3. 前端结构设计

```text
frontend/
├── public/
│   └── china.json
├── src/
│   ├── api/
│   │   └── index.js
│   ├── assets/
│   │   └── styles/
│   │       ├── theme.css
│   │       └── screen.css
│   ├── components/
│   │   ├── layout/
│   │   ├── cards/
│   │   ├── charts/
│   │   └── controls/
│   ├── composables/
│   ├── views/
│   │   ├── NationalOverview.vue
│   │   └── ShanghaiDetail.vue
│   ├── App.vue
│   └── main.js
├── package.json
└── vite.config.js
```

### 3.1 前端分层原则

- `views/`：页面级容器，只负责页面布局和数据编排
- `components/charts/`：ECharts 图表组件
- `components/cards/`：指标卡片、信息卡片
- `components/controls/`：滑块、回放按钮、筛选器
- `api/`：统一封装 Axios 请求
- `composables/`：复用状态逻辑，例如时间播放、图表自适应

### 3.2 页面映射

- `NationalOverview.vue` 对应 `Tab1`
- `ShanghaiDetail.vue` 对应 `Tab2`

## 4. 后端结构设计

```text
backend/
├── app.py
├── config.py
├── requirements.txt
├── extensions.py
├── db/
│   └── mysql.py
├── api/
│   ├── macro.py
│   └── shanghai.py
├── services/
│   ├── macro_service.py
│   └── shanghai_service.py
└── utils/
    └── response.py
```

### 4.1 后端职责划分

- `app.py`：Flask 应用入口、蓝图注册、CORS 初始化
- `config.py`：数据库连接、环境变量配置
- `db/mysql.py`：MySQL 连接封装
- `api/`：路由层，只做参数接收和响应输出
- `services/`：业务查询、聚合计算
- `utils/response.py`：统一 JSON 返回格式

### 4.2 接口归类

- `api/macro.py`
  - 全国概览
  - 全国城市地图
  - Top10
- `api/shanghai.py`
  - 指定小时站点流量
  - 单站点全天详情

## 5. 数据与脚本结构设计

```text
data/
├── count_df.csv
├── metro_city_flow.csv
├── station_info.json
├── processed/
│   ├── dim_station.csv
│   ├── fact_hourly_flow.csv
│   └── macro_city_flow_clean.csv
└── samples/

scripts/
├── clean_data.py
├── export_seed_csv.py
└── load_mysql.py
```

### 5.1 数据目录原则

- `data/` 根目录只放用户提供的原始数据
- `data/processed/` 只放脚本生成的标准化数据
- `data/samples/` 可放接口返回示例或开发期测试样本

### 5.2 脚本职责

- `clean_data.py`
  - 读取 `count_df.csv`、`metro_city_flow.csv`、`station_info.json`
  - 生成 3 张表对应的标准化 CSV
- `export_seed_csv.py`
  - 对中间数据进一步整理，便于导入和备份
- `load_mysql.py`
  - 把 `processed/` 中的数据导入 MySQL

## 6. SQL 目录设计

```text
sql/
├── schema.sql
└── seed_example.sql
```

约定：

- `schema.sql` 只放正式建表语句
- `seed_example.sql` 只放可选示例导入语句，不与建表混写

## 7. 文档目录设计

```text
docs/
├── 系统设计指南.md
└── 项目结构设计.md
```

说明：

- `系统设计指南.md` 负责讲系统目标、架构、接口、页面设计
- `项目结构设计.md` 负责固定目录和文件职责

后续如果继续扩展，可以新增：

- `接口文档.md`
- `部署说明.md`
- `数据字典.md`

## 8. 命名规范

### 8.1 文件命名

- Vue 页面组件：大驼峰，例如 `ShanghaiDetail.vue`
- JS 模块：小写或小写下划线，例如 `index.js`
- Python 脚本：小写下划线，例如 `clean_data.py`
- SQL 文件：语义化命名，例如 `schema.sql`

### 8.2 目录命名

- 使用英文小写
- 多单词使用连贯语义，不混用拼音和缩写

## 9. 后续开发落盘规则

后续我为你生成完整代码时，默认遵守以下规则：

1. 所有前端代码只写入 `frontend/`
2. 所有 Flask 服务端代码只写入 `backend/`
3. 所有数据库脚本只写入 `sql/`
4. 所有数据清洗和导入脚本只写入 `scripts/`
5. 不把业务代码直接堆在仓库根目录
6. 不额外引入你未指定的数据源
